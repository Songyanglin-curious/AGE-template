# AGENTS.md

## 项目意图

`<project-name>` 采用轻量级的吸引子引导工程工作流，用于 AI 辅助的应用程序开发。

此仓库属于应用层产品，而非框架核心项目。

仓库是真实来源，聊天仅是临时工作界面。

在编写重要代码之前，Agent 必须首先理解：

- `docs/context/project-context.md`
- `docs/context/ai-autonomy-policy.md`
- `docs/context/codebase-map.md`
- 项目上下文中列出的活跃需求
- 项目上下文中列出的活跃所有者文档
- 当需求含义依赖源材料时，`docs/input/` 下的相关原始输入

当事实冲突或不确定哪个工件拥有答案时，阅读 `docs/context/source-of-truth-and-precedence.md`。
当任务涉及规划或工作流决策时，阅读 `docs/process/application-development-workflow.md`。

## 任务路由

在编写代码之前，Agent 必须先对任务进行分类：

1. 确定任务类型：
   - 需求澄清
   - 应用层设计变更
   - 架构变更
   - 纯实现变更
   - 错误调查
   - 验证或审计工作
2. 行动之前，使用 `docs/index.md` 读取该任务类型的所有者文档。
3. 在起草或修订计划之前，检查 `docs/skills/README.md` 中的候选可复用技能。
4. 对于重要工作，在实现之前于计划中记录所选路由和计划使用的技能。

除非活跃需求和所有者文档已经明确了路由，否则不要直接从功能请求跳到代码。

## 操作规则

1. 优先采用文件输入、文件输出的协作方式。
2. 不要将聊天摘要视为持久的项目记忆。
3. 当范围仍不明确时，不要从原始的 PM 文本或原型截图直接跳到代码。
4. 如果输入模糊不清，应首先在 `docs/discussions/` 或 `docs/requirements/` 中创建或更新文件。
5. 当满足计划触发的条件时，在实现之前创建或更新计划。
6. 让 `docs/design/` 和 `docs/architecture/` 聚焦于当前支持的基线，而非迁移历史。
7. 保持日志简短、注明日期且仅追加。在完成任何重大代码变更后，你必须更新位于 `docs/logs/{year}/{month}-{day}.md` 的每日开发日志（按时间倒序排列，格式参考 `docs/logs/00-log-writing-guide.md`）。
8. 将非显式回归记录在 `docs/bugs/` 中。
9. 如果原型与实现存在重大分歧，请在 `docs/retrospectives/` 中记录原因，而不是悄然略过。
10. 只有当模式重复出现足以证明复用的合理性时，才将重复的过程经验提升至 `docs/skills/` 或 `docs/audits/`。
11. 对于高风险或高模糊性的需求、设计或计划草案，请请求独立的子代理或审查者进行审查，并反复修改直到解决重大反对意见。每个创建的计划在实现开始前都必须通过独立计划审计，并在标记为完成前通过独立关闭审计。
12. 保持代码注释最少。优先使用自解释的代码；只有当本地约束容易让人误解时才添加极少注释。
13. 当引用文件在预期路径中找不到时，在断定其不存在之前检查 `docs/archive/`。已归档的文件保留其原始相对名称于 `docs/archive/` 下。未经人工批准，不要将文件移至 `docs/archive/`。
14. 将可复用技能视为方法选择器，而非需求、设计或架构文档的替代品。业务知识应首先归属于所有者文档。
15. 当相同的错误模式反复出现时，不要止步于纯文本的经验教训。首先，在缺少相应方法时，将其提升为可复用的审计提示、检查清单或审查手册。如果缺陷模式仍然复发，则评估将其提升为启发式脚本、静态检查、代码规范规则、CI 守卫或代码转换工具，并根据所复制项目的实际约定和误报容忍度进行调整。

## 文档所有权

- `docs/context/` 拥有强制性的 AI 上下文、真实来源优先级和项目级约定。
- `docs/backlog/` 拥有优先排序的候选工作及可供 AI 执行的下一步行动。
- `docs/input/` 拥有原始外部输入，例如 PM 笔记、卡片文档、文章摘录、原型引用和复制的源材料。
- `docs/discussions/` 拥有需求澄清对话及未解决问题的记录。
- `docs/requirements/` 拥有可供实现的需求综合文档。
- `docs/design/` 拥有稳定的应用层业务和功能设计。
- `docs/architecture/` 拥有跨领域技术及模块边界的真实情况。
- `docs/lessons/` 拥有从错误、审计和回顾中提取的持久可复用经验。
- `docs/plans/` 拥有重要工作的执行和关闭标准。
- `docs/audits/` 拥有审计工作流记录及审计方法。
- `docs/skills/` 拥有可复用的提示模板、审查手册和审计提示模板。
- `docs/logs/` 拥有带有日期的实现记忆。
- `docs/testing/` 拥有手动和探索性测试记录。
- `docs/bugs/` 拥有非显式错误历史及回归说明。
- `docs/analysis/` 拥有研究、权衡分析及被否决的方向。
- `docs/retrospectives/` 拥有实现后差距分析及流程改进。

## 默认工作流

1. 在 `docs/input/` 中收集原始材料。
2. 如需澄清歧义，在 `docs/discussions/` 中进行。
3. 在 `docs/requirements/` 中综合形成可实施的需求。
4. 将稳定的设计输出拆分为 `docs/design/` 中的应用层设计和 `docs/architecture/` 中的技术设计，两者在需要时相互引用。
5. 分配任务并选择候选可复用技能。
6. 当计划触发条件适用时，编写或更新计划，并在相关时记录每阶段或每项的技能使用情况。
7. 实施前对计划进行审核。
8. 实现最小的完整切片。
9. 执行验证。
10. 对已创建的计划执行收尾审核。
11. 记录日志以及任何所需的缺陷笔记。

## 可选工作流层级

根据任务复杂程度按需使用。计划审核与收尾审核对于已创建的计划是强制性的。

- `docs/audits/` 用于文档审核及计划/收尾审核证据
- `docs/testing/` 用于手动或探索性验证
- `docs/retrospectives/` 用于填补材料需求或原型缺口
- `docs/skills/` 用于在多次失败后生成可复用提示
- `docs/lessons/` 用于在多次失败或重要恢复后记录持久工程经验教训

当工作需要同时在多个维度受到挑战时，使用 `multi-dimensional-audit-prompt.md`。当标准检查清单可能遗漏隐藏风险时，使用 `open-ended-audit-prompt.md`。这些提示是通用默认值，复制后必须根据项目的实际所有者文档、受保护区域、验证模型以及反复出现的故障模式进行定制。

## 规划规则

当任务具备以下任一特征时，需创建计划：

- 更改 API、数据库/模型、认证、集成、部署或公共契约行为
- 更改跨多个功能层面的用户可见行为
- 涉及多个模块并更改共享行为
- 预计需要超过一个 AI 会话
- 修改超过 5 个文件，或可能超过大约 200 行变更
- 需要分阶段执行或明确的收尾关卡
- 存在尚未解决的产品或技术风险，不应隐藏在聊天中

仅对局部低风险编辑（如文案修改、小型样式修复、仅测试清理以及已有明确测试的单文件行为修复）跳过正式计划。

所有创建的计划在实施开始前必须通过独立子代理或审核者审核，并在标记计划完成前再次审核。受保护区域、未解决的产品风险以及单一事实来源冲突需要人类/子代理审核或保持开放状态。

## 技能使用规则

在使用可复用技能之前，确认以下所有条件：

- 任务类型和路由已根据需求与所有者文档明确
- 技能与工作方法匹配，而不仅仅是业务标签相似
- `docs/skills/README.md` 中列出的所需输入可用
- 预期输出已知，并能存储在正确的文档位置

对于非简单计划，依赖可复用技能的每个阶段或条目都应记录 `Skill: <name>` 或 `Skill: none`。

## 面向代理的提示指导

- 不要根据单一功能列表生成完整产品。
- 不要以演示完整性为优化目标。
- 优先选择小的完整切片，而非广泛的占位覆盖。
- 优先采用现有项目模式，而非凭空发明的抽象。
- 如果信息缺失，将缺失的假设写入需求、讨论或计划文件中，而不是默默凭空补充。
- 不要将代码级别的实现细节放入计划文件，除非该细节对于范围或收尾推理是必需的。
- 优先引用现有的所有者文档，而不是在多个文件中重复同一规则。
- 不要将强制规则隐藏在 `docs/references/` 中；如果 AI 必须默认应用某项规则，将其放入 `docs/context/` 或 `AGENTS.md`。
- 使用 `docs/backlog/` 和 `docs/context/ai-autonomy-policy.md` 来决定 AI 是否可以在不询问的情况下选择并执行下一个任务。

## 文档维护

完成任何重大代码变更后，必须：

1. **更新每日开发日志**，位于 `docs/logs/{year}/{month}-{day}.md`（逆时间顺序，格式参见 `docs/logs/00-log-writing-guide.md`）。
2. **更新相关所有者文档**，位于 `docs/design/` 或 `docs/architecture/`，当变更影响应用层行为或技术结构时。

当验证完全通过（全绿），在日志条目中记录验证状态，并将其包含在 git 提交信息中。这为未来调试提供了可靠的已知良好基线。

## 验证基线

不要假设复制本项目时此模板的示例命令有效。

使用 `docs/context/project-context.md` 中列出的实际命令。

如果验证命令为空或仍为占位符，在报告验证成功之前先停下来将其填写完整。
