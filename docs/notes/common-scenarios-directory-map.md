# 常见问题场景 → 目录映射

## 用途

日常开发中遇到各种问题，不确定记录到哪个目录时，按此表查找。

---

## 场景索引

### 性能与故障

| 场景 | 放哪 | 说明 |
|------|------|------|
| 性能变慢，正在排查中 | `analysis/YYYY-MM-DD-performance-xxx.md` | 调查过程、火焰图截图、压测数据、阶段性结论 |
| 排查后确认是一个 Bug | `analysis/` 记过程 + `bugs/` 记根因 | 两份互补：`bugs/` 记根因和修复，引 `analysis/` 中的调查过程 |
| 排查后不是 Bug，是配置/环境/外部因素 | `analysis/` | 结论写清楚即可，不升格为 bugs |
| 线上故障，紧急修复 | `bugs/` | 根因 + 修复内容 + 预防措施 + 是否非显式回归 |
| 排查结果需要更新环境配合 | `analysis/` 记过程 → `lab/` 更新环境说明 | 避免后人踩同一个坑 |
| 仅仅是日常调优（加索引、加缓存） | `analysis/` | 优化前后对比数据、方案选择理由 |

### 需求与设计

| 场景 | 放哪 | 说明 |
|------|------|------|
| PM 给的需求相互矛盾 | `discussions/YYYY-MM-DD-topic.md` | 列出待澄清问题、每个问题的可能假设、等待确认的事项 |
| 原型只画了界面没写业务规则 | `discussions/` | 逐条列出缺失的规则，提出替代假设 |
| 需求写完了但还有残余问题 | `requirements/` 文件末尾的 `Open Questions` 节 | 非阻塞性的放这，阻塞性的不放这 |
| 第三方 API / 外部依赖突然变了 | `input/` 更新原材料 → `requirements/` 或 `analysis/` | 先记录变更来源，再评估影响范围 |
| 要做技术方案选型 | `analysis/YYYY-MM-DD-xxx-comparison.md` | 对比候选方案、记录权衡点、被否决方案及理由 |
| 选定方案后要改技术基线 | `analysis/` 记调研结论 → `architecture/` 更新系统基线 | 交叉引用，`architecture/` 是稳定版本 |

### Bug 与教训

| 场景 | 放哪 | 说明 |
|------|------|------|
| 修了一个明显的小 Bug（拼写错误、样式偏差） | 不单独记录，在 `logs/` 当天条目中提一句 | 微小修复不创建 bugs 文件 |
| 修了一个复杂 Bug（并发、数据不一致、边界条件） | `bugs/` + `logs/` | `bugs/` 记根因和预防，`logs/` 记简要 |
| Bug 根因揭示了一个流程缺陷 | `bugs/` 记 Bug → `retrospectives/` 记流程分析 | Bug 是点，Retro 是面 |
| 同一个错误模式在不同模块反复出现 | `lessons/` | 提取成持久工程教训（不再这么干） |
| 同一类问题反复发生，想建立检查清单 | `skills/` | 把教训升级为可复用的审计提示或检查清单 |

### 交付与回顾

| 场景 | 放哪 | 说明 |
|------|------|------|
| 交付后发现遗漏了重要需求 | `retrospectives/` | 分析缺口：缺了什么输入？哪个决策被推迟太晚？ |
| 原型和最终实现差距巨大 | `retrospectives/` | 分析原因：原型保真度低？需求中途变了？实现时发现问题？ |
| 需求文档写得很完整但实现走偏了 | `retrospectives/` | 分析：需求不可测试？实现时忽略了验收标准？ |
| 某个功能做完了想记录学到了什么 | `notes/` | 正面总结：这个过程中理解了什么新东西 |

### 学习与实验

| 场景 | 放哪 | 说明 |
|------|------|------|
| 跟着教程学了一个新库 | `notes/` | 结构化笔记，不是聊天记录复制 |
| 研究了一个设计模式怎么用 | `notes/` | 原理理解 + 在本项目中可能的应用场景 |
| 要搭一套本地测试环境 | `lab/` | 命令、配置、注意事项 |
| 要建一套性能压测环境 | `lab/` | 工具安装、脚本、数据准备步骤 |
| CI/CD 挂了不知道怎么修 | `analysis/` 记排查过程 → 修好后 `lab/` 更新配置说明 | 如果有人看管的配管，可能不在此层级处理 |

### 日常运转

| 场景 | 放哪 | 说明 |
|------|------|------|
| 今天干了什么（不论大小） | `logs/2026/06-08.md` | 简要条目 + 链接到详细文件 |
| 改了一个功能，涉及多个文件 | `logs/` 记结果 + `architecture/` 或 `design/` 更新基线 | 吸引子必须同步更新 |
| 不确定这算什么，先记下来 | `logs/` | 先记，以后判断是否需要迁移到专门目录 |
| 做了一个决定，怕以后忘了为什么 | `logs/` + 对应 `analysis/` 或 `retrospectives/` | `logs/` 记决定本身，详细理由放对应目录 |

## 覆盖盲区

以下场景 AGE 当前没有专门容器：

| 盲区 | 说明 | 临时处理 |
|------|------|---------|
| 团队协作问题（沟通失误、责任人缺失） | 属项目管理层面，AGE 不直接覆盖 | 可放 `retrospectives/` 作为流程缺陷分析 |
| 纯临时未成形想法（聊了一个 idea，还没想清楚） | AGE 有意不放这类半成品，避免仓库被碎片填满 | 先留在聊天；成形后进 `notes/` 或 `input/` |
| 部署/运维日常操作记录 | tools/ 和 notes/ 部分覆盖，没有专门 runbook 目录 | 放 `lab/` 或 `analysis/` |

## 判断口诀

```
在摸索 → analysis
是缺陷 → bugs
要总结 → lessons
缺什么 → retrospectives
学会了 → notes
环境呢 → lab
方法呢 → skills
今天呢 → logs
```
