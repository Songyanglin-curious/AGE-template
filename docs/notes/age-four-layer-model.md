# AGE 四层状态模型

## 概述

AGE 体系可理解为四层结构，它们有各自不同的状态机，而非共享一套统一状态流转。

---

## 第一层：宪法层（二态）

**角色**：底盘——AI 和人类在此仓库中的协作规则。

**文件**：`AGENTS.md`、`docs/context/`（`project-context.md`、`ai-autonomy-policy.md`、`codebase-map.md`、`source-of-truth-and-precedence.md`、`conventions.md`）

**状态机**：

```
[占位符/空] → [已填充]
```

**触发条件**：人类完成 `START-HERE-after-copy.md` 清单。

**流转规则**：此后宪法层**不被"经过"，而是被"读取"**。AI 进入任何阶段前先读它，但它不随工作项流转而改变自己的状态。仅当项目基本面变化（团队、自主权策略、受保护区域调整）时由人类更新。

---

## 第二层：吸引子层（三态）

**角色**：收敛目标——描述项目当前是什么、要往哪去。

**文件**：
- `docs/architecture/`：`project-vision.md`、`system-baseline.md`、`module-boundaries.md`
- `docs/design/`：`app-overview.md`、`feature-inventory.md`、`roles-and-permissions.md`
- `docs/requirements/`：`product-scope.md`、`mvp.md`

所有吸引子文件均为稳定文件（无日期，原地更新）。

**状态机**：

```
[新鲜] ⇄ [部分陈旧] → → → [陈旧]
  ↑                         ↓
  └──────── 对齐更新 ───────┘
```

**驱动力**：被执行链推着走——代码变更了但吸引子没更新 → `部分陈旧`；长期未维护 → `陈旧` 或 `未知`；关闭审计检查并更新吸引子 → `新鲜`。

**集中反映**：在 `project-context.md` 中以 `文档新鲜度` 字段体现，AI 据此决定自主行为：

| 新鲜度 | AI 行为 |
|--------|--------|
| `fresh` | 走正常流程 |
| `partially stale` | 仅已验证切片可写代码 |
| `stale` / `unknown` | 仅可研究，不能改产品行为 |

---

## 第三层：执行链（7 节点 + 2 闸门的主状态机）

**角色**：一件事从需求到落地的完整流水线。

**关键认知**：执行链不是 4 个平行子层，而是一条首尾相连的管道。

### 7 个节点

```
节点1: input/     状态: [已收集]
        │
        ▼
节点2: requirements/  状态: [草稿] → [可实施]
        │
        ▼
节点3: backlog/    状态: [idea] → [needs-requirement] → [needs-design] → [ready] → [in-progress] → [done]
        │ 条件: ready 且 AI自主权 = implement
        ▼
节点4: plans/      状态: [planned] → [in-progress] → [completed]
        │
        ▼
节点5: 代码仓库     状态: [未开始] → [进行中] → [已落地]
        │
        ▼
节点6: logs/       状态: [待记录] → [已记录]
        │
        ▼
节点7: testing/    状态: [待记录] → [已记录]
```

### 2 道闸门（强制审计，仅适用于创建了计划的工作项）

```
计划写完 → [闸门1：计划审计] → 通过 → 允许实现
                        → 不通过 → 修计划 → 再审

代码落地 → [闸门2：关闭审计] → 通过 → Backlog 标记完成
                        → 不通过 → 补漏洞 → 再审
```

### 三种规模的不同流转路径

| 规模 | 路径 | 特点 |
|------|------|------|
| 微小编辑 | 节点2 → 节点5 → 节点6 | 跳过 backlog、计划、审计、基准 |
| 中型功能 | 节点1 → 节点2 → 节点3 → 节点4 → [闸门1] → 节点5 → [闸门2] → 节点6 → 节点7 | 完整流程 |
| 大型跨会话 | 同上 | 节点4 多轮反复 [闸门1]，节点5 分阶段 |

### 各节点状态触发条件

| 节点 | 状态变更 | 由谁触发 | 触发条件 |
|------|---------|---------|---------|
| `input/` | 已收集 | 人 | PM 给了材料 |
| `requirements/` | 草稿 → 可实施 | AI | 验收标准明确，无阻塞性问题 |
| `backlog/` | 逐级推进 | AI | 每个前置条件满足（需求存在 → 设计基线存在 → 就绪） |
| `plans/` | 逐阶段推进 | AI | 每个阶段退出标准满足 + 审计通过 |
| 实现 | 代码提交 | AI | 分支合并 |
| `logs/` | 已记录 | AI | 每日结束后追记 |
| `testing/` 基准 | 已记录 | AI | 所有验证命令通过 |

---

## 第四层：经验库（无状态机）

**角色**：从项目过往中沉淀的可复用记忆。

**文件**：`docs/skills/`、`docs/lessons/`、`docs/notes/`、`docs/lab/`、`docs/analysis/`、`docs/bugs/`、`docs/retrospectives/`

**写入规则**：被动、只追加、不流转。在执行链某些环节**触发异常后写入**：

```
节点2 需求完成后 → 发现需求缺口巨大 → 写 retrospectives/
节点5 实现时     → 复杂 Bug，根因深 → 写 bugs/
关闭审计不通过   → 同类问题反复出现 → 写 lessons/ 或 skills/
技术选型时       → 需要深入调研     → 写 analysis/
```

**无"已解决""已完成"状态**——它不是任务，是记忆。写入后只读不修。

---

## 全局视角：四个时钟

```
宪法层：  ────────[已填充]──────────────→  改一次管很久

吸引子层：──[新鲜]──[部分陈旧]──[新鲜]──→  随执行节奏波动

执行链：  ─────────────────────────────→  连续流，不停
          input→reqs→backlog→plan→impl→log→baseline
               input→reqs→backlog→plan→impl→log→baseline

经验库：  ─────·─────·─────────·───────→  不连续，遇事才写
               bug  skill      retro
```

**总结**：宪法层不流转，只被读取。吸引子层被执行链推着更新。执行链是主状态机。经验库是执行链溢出的沉淀物。
