# AGE 系统如何运转

## 核心理念：先有吸引子，再谈控制

大多数 AI 开发方法论以"控制措施优先"——先问"怎么管住 AI？怎么验证？怎么记录？"

AGE 先问一个更前置的问题：

> **当人类和 AI 不断修改仓库时，它应该持续向什么方向收敛？**

这个收敛目标就是**吸引子**——落到仓库里的几份主人文档（Owner Docs）：
- `project-vision.md`：长期方向和不做什么
- `product-scope.md` + `mvp.md`：当前里程碑边界
- `design/`：当前支持的应用行为
- `architecture/`：当前支持的技术结构

有了吸引子之后，计划、测试、审计、日志这些**控制措施**才有意义——它们的任务变成：

> **证明某次变更确实把仓库带向了吸引子，而不是仅仅打勾任务清单。**

---

## 目录结构即操作系统

`docs/` 树被设计成项目的持久记忆系统，不是文档库，而是**文件输入/文件输出的运行时**。

按四层模型组织：

### 宪法层（底盘）
- `AGENTS.md`：AI 操作契约
- `docs/context/`：项目上下文、自主权边界、真相来源优先级、代码库地图

### 吸引子层（收敛目标）
- `docs/architecture/`：技术基线、模块边界
- `docs/design/`：应用行为、角色权限
- `docs/requirements/` 稳定部分：`product-scope.md`、`mvp.md`

### 执行链（流水线）
- `docs/input/`：原始 PM 材料
- `docs/discussions/`：歧义澄清（可选）
- `docs/requirements/`：综合可实施需求
- `docs/backlog/`：优先级 + 自主权标签
- `docs/plans/`：执行计划
- `docs/audits/`：独立质疑（计划审计 + 关闭审计）
- `docs/logs/`：逐日实现记忆
- `docs/testing/`：测试记录和已知良好基线

### 经验库（沉淀物）
- `docs/skills/`：可复用方法模板
- `docs/lessons/`：从失败中提取的工程教训
- `docs/notes/`：学习笔记
- `docs/lab/`：实验环境说明
- `docs/analysis/`：技术调研
- `docs/bugs/`：复杂回归根因
- `docs/retrospectives/`：实施后差距分析

## 执行链：10 步标准流程

### 1. 原始输入进 `input/`
PM 的飞书文档、Figma 原型、用户反馈——所有材料原样放入，不加工。保留模糊和矛盾。

### 2. 如有歧义进 `discussions/`（可选）
原材料模糊到无法写验收标准时，先澄清。

### 3. 综合成可实施需求（`requirements/`）
从原材料提取"现在要做什么"，写出**有具体验收标准**的需求。例："搜索 500ms 内返回，支持按姓名和手机号模糊匹配，空搜索显示全部，错误提示友好"。

### 4. 对齐吸引子
问：这个需求是当前基线已支持的，还是需要改变基线？
- 新功能：更新 `design/app-overview.md`、`feature-inventory.md`、`roles-and-permissions.md`
- 行为变更：更新受影响的主人文档
- 纯 Bug 修复且不改变行为定义：可能无需更新

### 5. 任务路由 + 技能选择
判断任务类型（需求澄清？设计变更？架构变更？纯实现？Bug 调查？验证？），然后检查 `docs/skills/README.md` 是否有可复用的工作方法。

### 6. 如果够大就写计划（`plans/`）
判断标准：涉及多模块、数据库/API/认证/集成/部署、预计跨会话、超 5 文件或 200 行代码 → 创建计划。

计划内容不是"实现细节"，而是：当前基线、目标/非目标、分阶段执行、每阶段退出标准、关闭关卡、技能选择记录。

微小编辑（改文案、改样式）跳过计划，直接快速路径。

### 7. 计划审计（强制）
独立子代理或审阅者质疑：范围定义诚实吗？关闭关卡可验证吗？有隐藏依赖吗？有未解决的需求缺口被掩盖吗？

通过才能写代码。不通过 → 修计划 → 再审。

### 8. 实现最小完整切片
实现端到端可工作的真实行为。宁可一个真实搜索页，不要五个空壳页面。

### 9. 验证
运行 `project-context.md` 中的真实验证命令。命令是占位符 → 不能报告验证成功。

### 10. 关闭审计 + 记录
独立的关闭审计：代码真落地了？文档对齐了？关闭关卡全满足了？没有本应完成的项目被悄悄降级为"以后再说"？

通过后：更新 `backlog` 状态 → 写 `logs/` → 如需更新 `known-good-baselines`。

## 两个控制钳

### 控制钳 A：自主权标签

`backlog/` 中每个工作项和 `project-context.md` 全局设置有一个自主权标签：

| 标签 | 含义 |
|------|------|
| `implement` | AI 读完主人文档可直接写代码 |
| `plan-first` | 只能先写计划，审计通过前不能写代码 |
| `ask-first` | 动任何代码前必须问人 |
| `research-only` | 只能查、看、分析，不能动代码 |
| `blocked` | 完全不能前进 |

**AI 只能单向收紧这些标签，不能放宽。** 防止越权。

### 控制钳 B：受保护区域

`ai-autonomy-policy.md` 中有受保护区域表（支付、数据删除、认证/权限等）。触碰到的任务其自主权被强制降级。

## 三种规模运转方式

### 微小编辑（改一行文案）
`requirements/` → 实现 → 验证 → `logs/`（不创建计划，不走审计）

### 中型功能（加一个搜索页）
`input/` → `requirements/` → `design/` 对齐 → 任务路由 → `plans/` → 计划审计 → 实现 → 验证 → 关闭审计 → `logs/` → `backlog` 更新

### 从零开始的大项目
- Phase 0：写 `vision.md` + `system-baseline.md` + `product-scope.md` + `mvp.md` + `app-overview.md`（每份不超过 1 页，确立吸引子）
- Phase 1：垂直切片（5-8 个端到端切片），写入 `backlog`，P0 排最高优先级
- Phase 2：对 P0 执行完整 AGE 循环
- Phase 3：P0 结束后根据新认知调整 backlog，再展开 P1 的详细需求，重复
- 只有当前片的细节是展开的，不提前做大设计

## 为什么这套系统有效

- **感知层**：输入不会在聊天中丢失
- **认知层**：模糊不会被假装成确定
- **稳态层**：收敛目标始终存在且可读
- **执行层**：每一步都知道是否在往吸引子走
- **验证层**："完成"不是自我宣称
- **进化层**：重复失败不会只留在记忆里

**核心目的不是让开发变快，而是让开发方向不会在多次 AI 会话之后散掉。**

---

## 关键规则速查

- 聊天是临时工作界面，仓库文件是持久记忆
- 稳定文件（描述当前基线）原地更新，带日期文件（记录历史）保留不动
- 档案移动必须人工批准
- 需求没有验收标准就还不能实施
- 计划的关闭关卡必须是可验证的，不能是"看起来做了就行"
- 审计必须是独立子代理或审查者，不能是自我宣称
