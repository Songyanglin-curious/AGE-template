# 真实来源与优先级

## 目的

本文件定义了哪个工件回答哪个问题。

使用它可避免混淆稳定事实、执行说明和历史上下文。

## 按问题的优先级

### 现在应该构建什么？

主要来源：

- `docs/requirements/`

支持来源：

- `docs/input/`
- `docs/discussions/`

规则：

- `docs/input/` 保存原始源材料
- `docs/requirements/` 是可实施的解释
- 如果它们不同，请显式更新需求文件，而不是悄悄依赖聊天记忆

### 当前支持的应用程序行为是什么？

主要来源：

- `docs/design/`

规则：

- `docs/design/` 拥有应用层功能、流程、角色和页面行为
- 功能需求文件可能驱动变化，但稳定的应用行为应汇聚到 `docs/design/` 下的所有者文档中

### 当前支持的技术结构是什么？

主要来源：

- `docs/architecture/`

规则：

- `docs/architecture/` 拥有技术边界、模块职责和横切实现规则

### 数据库事实是什么？

主要来源：

- 项目的数据库模型文件

示例：

- `model/`
- 模式 DSL 文件
- ORM 模型定义

规则：

- 数据库定义由模型/模式工件拥有，而不是由计划文本或散文文档拥有
- 文档可解释意图，但模型文件是真实来源

### API 契约事实是什么？

主要来源：

- API 模式文件、OpenAPI/GraphQL 定义、路由定义或后端契约测试

规则：

- 散文文档可总结 API 意图，但可执行或模式级别的 API 契约优先

### 外部集成事实是什么？

主要来源：

- 外部系统的集成契约文档
- 已提交的适配器配置或集成测试

规则：

- 不要仅根据 UI 需求臆造外部系统行为

### 环境/部署事实是什么？

主要来源：

- 部署清单
- 环境模式文件
- 基础设施配置

规则：

- 计划和文档可描述部署意图，但已提交的部署/配置工件是运营来源

### 此切片应如何执行和关闭？

主要来源：

- `docs/plans/`

规则：

- 计划是执行契约，而非长期所有者文档

### 执行期间实际发生了什么？

主要来源：

- `docs/logs/`

支持来源：

- `docs/testing/`
- `docs/bugs/`
- `docs/audits/`
- `docs/retrospectives/`

### 未来的 AI 会话应从重复失败中学习什么？

主要来源：

- `docs/skills/`
- `docs/lessons/`

规则：

- 使用 `docs/skills/` 存放可重用提示和操作手册
- 使用 `docs/lessons/` 存放可重用的工程教训和警示模式

## 冲突解决

- 如果原始输入和综合需求不一致，请在编码前更新 `docs/requirements/` 或重新开启澄清。
- 如果需求与所有者文档不一致，决定需求是否更改了支持的基线；然后显式更新 `docs/design/` 或 `docs/architecture/`。
- 如果实时代码与所有者文档不一致，将其视为实现偏差或陈旧的文档；不要默默选择其一。
- 如果解决冲突会改变用户可见行为、数据/模型形态、API 行为、认证/权限行为或外部集成行为，请停止并请求确认。
- 如果验证失败，即使实现看似完成，计划也不得关闭。
- 如果模式/模式文件与散文文档在数据库事实上不一致，模式/模式文件优先；有意地更新散文文档或模型。

## 遗留或陈旧文档模式

当 `docs/context/project-context.md` 将当前切片的文档新鲜度标记为 `stale`、`unknown` 或 `partially stale` 时使用此模式。

- 实时代码和可执行契约是当前行为的证据，而非自动期望的行为。
- 所有者文档仅在与实时代码、需求以及人类/产品意图重新验证之后，才成为预期目标。
- 在更改行为之前，在需求、讨论、分析或计划文件中将每个冲突分类为 `implementation drift`、`doc drift` 或 `intentional legacy behavior`。
- AI 自主权默认为 `research-only` 或 `plan-first`，直到基线审计或人类确认记录应保留与应更改的内容。对于 `partially stale`，此限制仅适用于其需求、所有者文档、代码库映射路由或触碰的代码区域尚未验证为最新的切片。
- 不要在没有记录偏差分类的情况下“修复”代码以匹配陈旧文档，或重写文档以匹配代码。

## 简单的经验法则

- 稳定的行为和结构属于所有者文档
- 执行属于计划和日志
- 历史和诊断属于错误报告、审计、测试笔记、回顾和经验教训
