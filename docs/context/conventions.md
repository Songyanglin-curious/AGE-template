# 项目约定

## 目的

此文件记录了 AI 代理默认应遵循的项目级规则。

保持简短。如果某条规则成为详细的参考资料，请将详细内容移至 `docs/references/` 并在此处链接。

## 文件输入/输出

- 重要的输入应在实施前写入文件。
- 重要的输出应写回仓库，而非仅保留在聊天记录中。
- 原始输入应放入 `docs/input/`。
- 综合整理后的可实施需求应放入 `docs/requirements/`。

## 设计划分

- 需求/应用行为设计应放入 `docs/requirements/` 和 `docs/design/`。
- 技术架构设计应放入 `docs/architecture/`。
- 通过交叉引用避免在多个文档中重复相同的规则。

## 审查规则

- 高风险或高模糊性的需求和设计草案应由独立的子代理或审查者进行审查。
- 每个创建的方案在实施前需经独立的方案审计，完成后需经关闭审计。
- 独立审查应引用具体文件和证据，而不能仅说“看起来不错”。
- 如果无独立审查者可用，请在方案或日志中记录此限制。冷重放不是第二审查者，绝不自行解决受保护区域或真实数据源的冲突。

## Bug 规则

- 每次非微小 Bug 修复都应增加或更新自动化测试覆盖率。
- 若无法实现自动化覆盖，请记录原因和人工验证证据。

## 注释策略

- 默认情况下倾向于不写注释。
- 仅当局部约束易被误读且仅靠代码本身不足以说明时，才添加注释。

## 验证规则

- 保持 `docs/context/project-context.md` 中的验证命令为最新。
- 对于未实际运行的命令，不得报告验证成功。
- 复制模板后，不得保留占位验证命令。
