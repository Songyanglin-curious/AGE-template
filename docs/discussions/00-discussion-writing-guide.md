# 讨论文档编写指南

## 用途

使用 `docs/discussions/` 记录针对模糊工作的多轮澄清。

## 适用场景

- 产品经理只能提供部分确认
- 原型含义不明确
- 仍存在多种需求解读可能
- 开发人员原本可能会直接从原始文件中推断领域规则

## 包含内容

- 被讨论的源文件
- 待解决问题
- 候选解读
- 已确认的决策
- 阻碍实现的未决事项

## 规则

讨论文件用于澄清目的，并非最终事实。应将已确定的结论移入 `docs/requirements/`、`docs/design/` 或 `docs/architecture/`。

## 文件命名指南

推荐使用带日期的文件名：

- `docs/discussions/YYYY-MM-DD-topic.md`
