# 审计执行指南

## 目的

本指南定义了应用层开发的默认审计检查点。

计划审计和结项审计对于已创建的计划是强制性的。

每个已创建的计划必须在至少一处持久记录审计证据：计划、每日日志或 `docs/audits/` 下的文件。不要仅在聊天中保留审计结果。

冷回放不是第二位审查者，不能单独批准计划创建、计划结项或受保护区范围的变更。

## 三种默认审计

1. 文档审计
2. 计划审计
3. 结项审计

这些是审计对象。

项目也可以在这些对象上应用审计风格：

- 多维度审计——从多个维度对工作进行质疑
- 开放式审计——在标准检查清单之外搜索隐藏问题

示例：

- 多维度计划审计
- 开放式结项审计

模板在 `docs/skills/` 下为这些风格提供了通用的默认提示。复制的项目应将这些提示调整至自身的所有者文档、受保护区、验证堆栈及已知失败模式。

## 文档审计

在需求/设计更新之后、大规模实现工作之前执行。

对于高风险、跨模块或跨文档的工作，考虑在常规文档审计之上叠加 `multi-dimensional-audit-prompt.md`。

检查内容：

- 缺失的范围边界
- 伪装成已确定需求的未决问题
- 原始输入与综合需求之间的不匹配
- 需求与所有者文档之间的不匹配

## 计划审计

在编写计划之后、实现之前执行。

如果计划跨越多个所有者文档边界、受保护区或验证层面，添加 `multi-dimensional-audit-prompt.md`。

检查内容：

- 不诚实的结项关口
- 隐藏的依赖项
- 没有所有者的遗留事项
- 仍依赖未解决需求的计划范围
- 缺少每个验收标准的证明策略
- 计划中缺失或薄弱的任务路由和技能选择理由

## 结项审计

在每个已创建计划的实现和验证之后执行。

若常规结项检查持续通过，而之后仍出现隐藏回归或证明薄弱，则添加 `open-ended-audit-prompt.md`。

检查内容：

- 实际行为确实已落地
- 文档已对齐
- 所声称的证明确实存在
- 计划结项关口真正得到满足
- 未将范围内条目降级为模糊的后续工作
- 验证失败未在无明确裁决的情况下被视为非阻塞

## 输出规则

对于每个已创建的计划，审计结果必须持久记录。当审计具有非平凡性、存在争议、可复用或可能之后重要时，使用单独文件。

对于可复用的提示模板，使用 `docs/skills/`；对于附加到每日执行中的小型审计笔记，使用 `docs/logs/`。

## 文件名指导

审计记录优先使用带日期的文件名：

- `docs/audits/YYYY-MM-DD-document-audit-topic.md`
- `docs/audits/YYYY-MM-DD-plan-audit-topic.md`
- `docs/audits/YYYY-MM-DD-closure-audit-topic.md`
