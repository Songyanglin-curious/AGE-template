# 从规范驱动开发到吸引子引导工程

`nop-chaos-flux` 是一个低代码运行时框架，基于 React 19、Zustand 5 和 Vite 8 构建。它受百度 AMIS 启发，但围绕创新设计原则从零开始独立重新实现。它包括一个 JSON Schema 编译器和运行时，以及报表设计器、流程设计器等设计工具。

该项目的初始版本由一名程序员在两个月内完成，完全通过自动化的 AI 辅助开发。与普通的 AI 辅助开发不同，该项目并没有随着功能增加而出现质量下降。相反，模块边界变得更加清晰，文档分类更加稳定，测试和审计在捕获实际问题方面做得更好，代码质量持续提升。

我最近做了几次内部分享，并将这种实践命名为 **吸引子引导工程**（AGE）。后来在代码评审中发现，许多人还是没有真正理解它。有些人的工作流程接近于随意的 vibe coding。另一些人使用类似 OpenSpec 风格的规范驱动开发，整体倾向于任务驱动、逐功能执行。随着工作积累，代码中的不一致也不断累积，偏离架构设计成为常态。

本文分析了 AGE 与规范驱动开发之间的区别，以及普通业务项目可以从 `nop-chaos-flux` 中的大规模 AI 工程实践中借鉴什么。

## 核心层次结构

AGE 的核心是以下概念层次：

**状态空间 -> 吸引子 -> 轨迹 -> 控制**

- **状态空间**：系统在当前约束下可能演变成的所有实现状态
- **吸引子**：系统在长期演化中被反复拉回的稳定结构
- **轨迹**：每一轮生成、验证和修正后留下的实际演化路径
- **控制**：通过局部信号持续影响轨迹的机制

在现有约束下，仓库可能演变成的代码、文档和测试的所有可能组合构成了状态空间。人类、AI、评审、CI、文档更新以及其他持续进行的操作形成了演化规则。它们共同产生实时的仓库历史，这就是轨迹。吸引子则是系统在长期迭代中被反复拉回的那个稳定结构。

大规模 AI 开发本质上是一个动力系统中的受控收敛问题。AI 极快地扩展了状态空间。关键不是到处设置护栏，而是首先要理解系统长期应该被拉回什么样的结构。

吸引子不是一个写死的终点，不是路线图，也不是允许活动的边界。它是由少量高层约束隐含定义的结构：局部实现可以有差异，但整体会不断被这些关系拉回同一种形态。

在 `nop-chaos-flux` 中，最先定义吸引子的不是计划、lint 规则、测试、审计或 spec 文件，而是 `docs/architecture/` 下具有明确优先级的架构文档。在下文中，这些具有所有者和优先级的架构文档称为所有者文档。

`docs/architecture/README.md` 明确指出，`docs/architecture/` 是当前的最终状态架构基线，`flux-design-principles.md` 是治理原则锚点，`frontend-programming-model.md` 拥有顶层原语和核心边界的最高优先级，其他规范性文档在其主题范围内拥有局部优先级。

这些文档并不枚举每一种正确的实现。它们像方程一样，定义了一个稳定结构：哪些原语存在，哪些依赖方向是合法的，哪些所有者边界不能打破，哪些旧模式已不再属于正确的状态空间。它们首先定义了可持续的状态集群；此后，实现、测试、规划和审计才有了参照点。

计划、验证、审计、日志、缺陷和测试都排在后面。它们不是吸引子本身，而是让系统轨迹靠近吸引子的约束套件。

Flux/AGE 并非以 spec 演化为中心，而是以吸引子定义和轨迹收敛为中心。

## 实际开发历史

“架构文档定义吸引子”乍一听可能像是普通的文档治理。但如果你仔细查看 `docs/logs/` 下的实际开发记录，区别就显而易见了。

例如，`docs/plans/371-deep-audit-2026-05-19-owner-routing-plan.md` 是一个典型的入口点。它没有直接编辑代码，而是将 `2026-05-19` 深度审计中保留的 64 个发现收敛成一个所有者路由基线。每个发现都有唯一的所有者桶、优先级、后继计划和所有者文档义务。不存在无主问题、无交叉归属问题，也绝不会被悄然降级为含糊的待办事项。

它的作用是冻结一轮偏差应如何关闭。随后的执行计划再将局部轨迹收敛到具体的结果面。`docs/plans/382-deep-audit-2026-05-19-table-and-crud-owner-state-and-event-contract-plan.md` 关闭了表格/CRUD 所有者状态和事件负载：`Current Baseline` 声明了显式空数组所有者状态和事件负载的当前状态，执行条目修复了空数组回退，聚焦的测试证明了结果，`docs/components/table/design.md` 和 `docs/components/crud/design.md` 同步了所有者契约，最终全仓库的 `pnpm typecheck`、`pnpm build`、`pnpm lint` 和 `pnpm test` 全部通过。独立的关闭审计 `ses_1bd9ed593ffeVpkho4lb4wPR6p` 记录了 `Verdict: acceptable`。

`docs/plans/388-deep-audit-2026-05-19-form-tree-widget-accessibility-plan.md` 展示了另一种闭合方式。输入树和树选择可见节点的游走焦点、`ArrowUp` / `ArrowDown` / `Home` / `End` 导航以及加载 `aria-busy` / `aria-describedby` 都已具备聚焦证明，但该计划明确裁定了 `No owner-doc update required`。这表明计划不仅仅是任务列表。它还负责裁定该变更是否修改了所有者文档基线。

`docs/plans/400-deep-audit-2026-05-19-test-harness-reliability-plan.md` 和 `docs/bugs/62-e2e-shared-websocket-error-suppression-fix.md` 展示了记忆系统的作用。问题不在于某个业务功能失败。端到端共享夹具过滤范围过宽，同时也压制了真正的传输/运行时失败。修复不仅仅是更改代码。它还添加了 `tests/e2e/fixtures-hard-gate.spec.ts` 来证明该夹具现在会失败，并记录了一条错误备注规则：如果未来的测试确实预期 WebSocket 失败，它必须使用逐测试许可，并且不得恢复夹具范围的压制。

`docs/logs/2026/05-19.md` 和 `docs/logs/2026/05-20.md` 不是日记。它们为每个闭合切片记录了聚焦证明、所有者文档裁定、仓库级门禁、独立审计和全绿基线。

这些材料构成了一个真实的链条：所有者文档定义长期结构；审计发现偏差；计划对其进行路由并冻结所有权；聚焦证明展示局部结果；所有者文档裁定决定基线是否改变；日志、缺陷记录和测试保留跨会话记忆；闭合审计从活跃仓库重新判断完成情况。

这就是 AGE 中的基础框架。目标不是让每个任务看起来更完整，而是保持系统轨迹回归吸引子。

## OpenSpec 的结构性优势

OpenSpec 是一个代表性的规范驱动开发框架。它很好地结构化了一类工作：

- `openspec/specs/` 存储能力的当前行为规范。
- `openspec/changes/` 存储提议的变更，包括 `proposal.md`（为何变更、变更什么、影响范围）、`design.md`（必要时提供技术设计）和 `tasks.md`（实施清单）。
- `changes/<name>/specs/` 使用 `ADDED`、`MODIFIED`、`REMOVED` 和 `RENAMED` 记录规范增量。
- `archive` 应用固定规则将增量写回主规范。

这种机制降低了更新规范的成本。OpenSpec 不需要 AI 每次都自由地理解整个需求，然后随意编辑文档。相反，它使用固定的章节、需求标题匹配和增量应用，将结构化变更合并回主规范。当行为规范明确且需要可解析、可归档并一致写回时，这种方法效果很好。

## OpenSpec 的局限性

OpenSpec 的规范格式包含需求、场景、SHALL/MUST 和增量部分，对工具友好。但作为组织项目知识的通用方式，它不够灵活。

许多仓库知识天然不适合需求/场景的形式：

- 架构层次与优先级
- 术语误读的历史
- 外部研究材料，如 PPT 和讨论记录
- 源代码审计的结论
- 手动测试的观察结果
- 复杂错误的诊断路径
- 某个计划未能闭合的原因

如果所有这些都必须先转换为规范/变更形式才能获得正式位置，转换成本高且原始上下文丢失。外部文档、讨论记录、分析报告和缺陷回顾可以直接成为仓库记忆。强行将它们塞入需求/场景反而可能扁平化信息。

在执行验证方面，这两种方法也反映了不同的工具哲学。OpenSpec 的 `tasks.md` 是一个清单，而 `/opsx:verify` 是一个可选的代理验证技能，必须通过扩展工作流手动启用。`archive` 检查任务状态并对风险发出警告，但用户确认后仍可继续。严格的闭合留给团队处理。

Flux 计划将独立的闭合审计内建到必要路径中。一个计划必须从当前基线开始，陈述 `Goals` 和 `Non-Goals`，定义执行项，每个项标记为 `Fix` / `Decision` / `Proof`，并指定 `Closure Gates`。在标记为 `completed` 之前，独立的子代理或审核者必须回到活跃仓库并执行闭合审计。`docs/plans/361-slot-contract-host-manifest-and-owner-doc-closure-plan.md` 将当前基线、调查结果所有权矩阵、执行阶段、闭合门禁、经裁定的延期项和独立审核证据整合在一起。这不是清单。它严格定义了完成条件并要求独立的审计证据。**它强调的不是完成了什么，而是某件事必须达到何种程度才算真正算数。**

## 为什么规范驱动开发容易变成任务导向

规范驱动开发自然地将人和 AI 的注意力拉回“如何完成这个变更？”。

一个变更通常包含提案、设计、任务和增量规范。这比口头需求好得多，但它天然围绕单个变更组织：为何变化、变更了什么、如何完成、哪些清单项已完成。久而久之，规范驱动开发容易变成更正式的任务分派系统。

这种任务导向有一个隐藏的风险：规范已更新，任务已检查，归档已运行，AI 和人类都产生了强烈的完成感。但系统是否真的更接近其长期结构，并非由变更本身证明。这必须对照架构文档、活跃仓库、测试、日志和独立审计来检查。

如果您试图以 Flux 级别的精度维护规格，成本通常会更高。Flux 并不维护单一的规格树。它维护一组具有不同职责的仓库记忆：架构文档定义吸引子；计划定义局部轨迹闭合；日志记录演进过程；缺陷保留复杂诊断；测试保留人类发现；分析保留研究判断；审计重新检查活仓库。将所有这些强行塞入一个规格/变更结构，会持续产生转换成本。

在 AGE 内部，规格驱动开发可以是一个用于管理行为规格演进的本地 harness。但它不应该取代架构文档，也不应该取代计划闭合、日志、缺陷、测试和审计这些独立的仓库记忆。

## 如果您只需要阅读一份 Owner 文档？

这体现了两种方法之间的根本区别。

在 Flux 中，如果您只需要检查渲染器运行时的当前 owner 契约，可以直接从 `docs/index.md` 路由到 `docs/architecture/renderer-runtime.md` 及相关引用。您不需要创建变更、编写提案、定义任务，或将问题重述为规格增量。

如果只是研究，可能进入 `docs/analysis/`。如果是手动测试发现，进入 `docs/testing/`。如果是复杂缺陷，进入 `docs/bugs/`。如果是执行闭合，则编写计划。如果稳定基线发生变化，则返回 owner 文档。

Flux 的文档组织是自由的，但并非随意。自由意味着它不强制所有类型的知识都首先通过相同的工作产物流程。受限意味着每种素材类型都有明确的职责，不能随意宣称事实权威。

OpenSpec 的组织更结构化。它适合将提议的行为变更放入变更包，并将规格增量写回主规格。但路由通用仓库知识并不是它的默认主线问题。

## 任务不是计划

OpenSpec 的 `tasks.md` 并不对应 Flux 的 `plans`。

至多，`tasks.md` 对应 Flux 计划内部的执行列表片段，或当前会话的待办事项列表。它有助于 AI 避免遗漏步骤，但无法回答以下问题：

- 当前活仓库基线是什么？
- 本次变更中明确不做哪些工作？
- 什么证据证明工作真正完成？
- 哪些缺陷不能降级为后续处理？
- 谁独立审核完成情况？

Flux 计划的价值不在于它包含更多任务。它是局部轨迹的闭合契约。

因此任务对 AI 有用，但任务并非事实来源。`- [x]` 仅意味着实现者声称某项已完成。真正的证明必须回到当前代码、测试、owner 文档和独立审核。

## 变更来源不仅是规格变更

OpenSpec 有一条清晰路径：围绕规格和变更来组织行为规格的演进。

Flux 的变更来源更广泛：

- 用户提出的新需求
- 阅读源代码发现的架构漂移
- 审计发现的 owner 边界问题
- 测试问题暴露的交互问题
- 缺陷记录中保存的历史回归风险
- 外部研究或分析得出的设计判断
- 计划闭合审计推翻先前“已完成”的判断

这些来源不一定首先要转化为规格增量。它们可能进入分析、测试、缺陷、日志、计划，或直接更新架构文档。它们是否需要成为行为契约，取决于它们是否属于长期基线的一部分。

这正是 Flux 文档组织自由的意义所在。不同来源的信息首先存放在不同位置，而不是全部塞入一个规格演进通道。

## 为什么 File-In / File-Out 重要

我在演讲中多次强调，AI 驱动开发的最佳实践之一是 file-in / file-out。这不仅仅是为了文档整洁。吸引子和 harness 需要载体。

File-in：输入不应只留在聊天窗口。即使内容杂乱，先将其写入需求文件、分析文件或计划文件，然后在 OpenCode 中 `@` 该文件。

File-out：输出不应只在窗口打印然后消失。重要的结论、分析、计划、测试记录、缺陷诊断和架构约束都应写入 `docs/`，并按职责分类：

- 架构规则 -> `docs/architecture/`
- 变更闭合 -> `docs/plans/`
- 执行轨迹 -> `docs/logs/`
- 手动测试发现 -> `docs/testing/`
- 复杂缺陷诊断 -> `docs/bugs/`
- 研究判断 -> `docs/analysis/`

重点不是写更多文档。一旦输出被分类并持久化，它就进入仓库记忆。聊天窗口是临时上下文。文件是仓库记忆。

## 代码是当前实现事实，而非唯一事实

代码是当前实现的事实来源。类型和测试保护当前行为。具有优先级的架构 owner 文档定义吸引子：系统长期应趋向的结构。日志、缺陷、分析和测试记录轨迹与外化记忆。

这些并不冲突。冲突发生在两种情况下：

- 文档开始重复脆弱的实现细节
- 将文档的完成视为代码已正确的证明

OpenSpec 的行为优先边界也试图避免第一个问题：行为规格应描述可验证的行为，而非内部实现细节。Flux 则进一步强调第二个问题：即使文档陈述了正确行为，它也不能取代当前仓库验证。真实的闭合必须回到代码、测试、owner 文档、日志和审计证据。

## 真正的区别

OpenSpec 解决的是这个问题：如何通过结构化的规格、变更、变化量（delta）和归档来组织行为规格的更新。它的核心是规格的演化流程。

AGE 解决的是这个问题：当 AI 高速扩展状态空间时，系统如何先定义吸引子，再利用计划、验证、审计、日志、缺陷、分析、测试等约束手段，让轨迹持续收敛？

两个工具在不同层面解决问题。OpenSpec 擅长结构化的规格工作流。Flux/AGE 擅长通过具有优先级的架构所有者文档定义吸引子，再使用自由但责任有界的文档路由和严格的闭合约束，使仓库随时间推移持续收敛。

从规格驱动开发转向吸引子引导工程，不是从“写规格”退回到“不写规格”，而是从“将变更组织为规格更新”转向“定义系统长期应收敛到哪里，并将每一轮 AI 生成不断拉回那个结构”。

## 普通项目可直接使用的内容

你不需要完整复制 Flux 的治理体系。

**第一，文件入/文件出。** 在将输入交给 AI `@` 之前先放入文件。把输出写入 `docs/` 而不是只留在聊天窗口里。这只需几分钟，却能让 AI 依赖可追溯的仓库记忆而非上下文记忆。

**第二，为核心业务撰写架构文档并定义业务吸引子。** 不要写代码导览。要写事实来源、状态转换、所有权边界和常见误解。

**第三，为复杂变更编写轻量级计划。** 包含当前基线、目标、不做的事项、执行项、证明项和闭合条件。在完成前，使用独立的会话或独立审阅者进行复核。

**第四，将重复出现的问题转化为仓库记忆。** 手动问题进入测试，复杂缺陷进入 bugs，研究结论进入 analysis，高频错误进入审计脚本。

规格更新是必要的一层。系统的收敛仍然需要吸引子。
