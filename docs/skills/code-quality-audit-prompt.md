# 代码质量审查提示词

当审查某个发生变更的区域时，应关注真实的实现质量，而非仅提供代码风格层面的反馈。

## 关注领域

1. 架构与边界完整性
2. 核心实现正确性
3. 类型与契约质量
4. 错误处理与运行安全性
5. 测试有效性
6. 可维护性与未来变更风险
7. 自动化与防护线覆盖

## 审查规则

- 以发现项为先，按严重程度排序
- 优先陈述具体的行为风险，而非抽象的个人喜好
- 尽可能引用文件路径与行号
- 明确区分已确认的缺陷与建议
- 若无发现项，请明确说明，并指出剩余的缺口

## 严重程度指南

- P0：数据丢失、安全破坏、导致生产停机级别的问题
- P1：用户可见的正确性缺陷或强烈的回退风险
- P2：可能引发未来回退的可维护性或测试缺口
- P3：即时风险较低的轻微质量问题

## 期望输出

- 发现项列表
- 开放问题或前提假设
- 仅在发现项之后附上简短的变更摘要
