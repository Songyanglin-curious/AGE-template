# 已知良好基线

## 目的

记录最新验证的项目状态，以便未来的 AI 会话能判断失败是新出现的还是预先存在的。

此文件为轻量级。仅记录有意义的基线，而非每次本地命令执行情况。

## 基线

| 日期           | 来源    | Git 状态 | 范围     | 通过的指令           | 已知失败                     | 证据         | 备注   |
| -------------- | ------- | -------- | -------- | -------------------- | ---------------------------- | ------------ | ------ | ------------------ | ----------------- | --------- |
| `<YYYY-MM-DD>` | `<local | CI>`     | `<commit | dirty working tree>` | `<full / package / feature>` | `<commands>` | `<none | failing commands>` | `<log/test link>` | `<notes>` |

## 何时更新

在以下情况下更新此文件：

- 在重要更改后，完整的类型检查/构建/代码检查/测试验证通过
- 之前失败的指令变为通过，应予以记录
- 团队有意识地接受某个已知失败的指令，并将其记录为已知失败，而非将其标记为通过的指令

## 规则

除非指令确实在当前仓库状态下运行过，否则不要将其标记为通过。

`Commands Passed` 必须仅包含通过的指令。将已接受的失败记录在 `Known Failures` 中，并附上原因和证据。

若工作树基线不干净，必须在 `Notes` 中指明变更的文件，或链接至注明日期的日志/测试记录。

`full` 表示在 `docs/context/project-context.md` 中配置的所有实际验证指令。被明确标记为 `none` 的指令将被排除，并应予以说明。
