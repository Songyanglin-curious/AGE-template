# 实施指南

## 目的

本指南记录了面向 AI 和人类的日常编码期望。

## 核心规则

- 优先选择最小的正确变更
- 遵循现有模式，然后才发明新的抽象
- 保持状态所有权明确
- 避免使用仅用于演示的占位符，除非当前需求明确允许
- 当缺陷或约定可能静默回归时，添加测试
- 除非为范围、证明或约定推理所必需，否则不要将代码级技术细节放入计划文件中
- 默认不写注释；仅在局部约束容易误解时添加少量高价值注释

## 审查视角

审查工作时，检查：

- 边界违规
- 未写入文件的隐藏假设
- 仅基于 UI 形态或聊天声称的虚假完成
- 缺少对用户可见行为的验证

## 协作规则

如果重复的指令在聊天中反复出现，将其提升为 `AGENTS.md`、`docs/architecture/` 或 `docs/references/`。

如果重复的审查模式不断发现同类问题，将其提升为 `docs/skills/` 或 `docs/audits/`。

如果此后相同的缺陷模式不断重现，评估是否应将其进一步提升为启发式脚本、静态检查、lint 规则、CI 守卫或 codemod。优先选择轻量级检查，并针对所复制项目的实际约定、命名模式和误报容忍度进行调整。
