# 目录导航指南

## 用途

快速理解 `AGE-template` 每个目录和根文件是干什么的，该看哪个。

---

## 名词对照

AGE 方法论有一套专有名词。本文中名词后跟 `（简短说明）` 帮助理解，等后续词语稳定后再统一替换。

| AGE 名词 | 简短说明 |
|----------|---------|
| 吸引子 | 项目收敛目标，当前可信主线 |
| 轨迹 | 推演过程、执行记录、误判修正 |
| 轨迹挖掘 | 从历史过程里提炼规则和经验 |
| 路由机制 | AI 读取文档的入口和索引规则 |
| 所有者文档 | 某模块/规则/需求的权威说明 |
| 控制措施 | 计划、测试、审计、日志等工程控制手段 |
| 自主权标签 | AI 可否自行行动的权限级别（implement/plan-first/ask-first 等） |
| 受保护区域 | 触碰时权限必须降级的高危区域（支付、权限等） |
| 执行链 | 需求到代码落地的完整流水线 |
| 经验库 | 从项目过往中沉淀的可复用记忆 |
| 计划审计 | 实现前的独立验证关卡 |
| 关闭审计 | 完成后的独立验证关卡 |
| 文档新鲜度 | 文档与代码的一致性状态标记（fresh/stale 等） |
| 已知良好基线 | 全部验证通过的可信快照记录 |
| 冷重放 | 搁置上下文后的文档化自检，非第二审查者 |
| 宪法层 | AGENTS.md + docs/context/ 构成的底盘规则 |
| 最小完整切片 | 端到端可工作的最小功能单元 |
| 稳定文件 | 原地更新、描述当前基线的文件（无日期） |
| 时效文件 | 带日期、记录历史过程的文件 |

## 根级文件

### `README.md`

**用途**：模板总览。AGE 设计理念、适用范围、项目结构、核心原则、推荐的执行模式。

**什么时候看**：第一次接触这个模板的时候。之后日常不需要反复看。

### `AGENTS.md`

**用途**：AI 代理操作契约。规定了 AI 在仓库中读什么、什么时候可以自己动、什么时候必须停。

**什么时候看**：AI 每次会话开始时应逐条检查。人类不定时翻阅了解 AI 行为边界。

## 宪法层（底盘规则）

### `docs/context/`

**用途**：AI 代理在进行非微小变更之前必须读取的强制性上下文。**每个 AI 会话的起点。**

| 文件 | 内容 |
|------|------|
| `project-context.md` | 当前项目身份、技术栈、活跃需求、验证命令、文档新鲜度、自主权级别 |
| `ai-autonomy-policy.md` | AI 何时可自行行动、何时必须停、受保护区域定义 |
| `codebase-map.md` | 仓库入口点、常见变更路径、易出错文件清单 |
| `source-of-truth-and-precedence.md` | 每类问题该信哪个文件（需求信需求文档、数据库信模型文件、API 信契约文件等） |
| `conventions.md` | 项目级约定（文件输入/输出、设计划分、审查规则、注释策略） |

**关键规则**：放规约和强制上下文，不放长篇文章和日常记录。

### `docs/process/`

**用途**：开发工作流文档——说明项目应该如何被开发。

**入口**：`application-development-workflow.md`，定义了 13 个阶段的完整流程（input → requirements → design → plan → audit → implement → verify → closure audit → logs）。

**什么时候看**：第一次执行多阶段任务前，或不确定当前处于哪个阶段时。

### `docs/references/`

**用途**：辅助查找的参考工具。不定义强制规则（强制规则在 `docs/context/` 或 `AGENTS.md`）。

| 文件 | 内容 |
|------|------|
| `maintenance-checklist.md` | 代码变更后需更新哪些文档 |
| `document-naming-and-timeliness.md` | 哪些文件用稳定名、哪些用带日期名 |
| `implementation-guide.md` | 实施参考 |
| `directory-guide.md` | 本文件 |

## 吸引子层（收敛目标）

所有吸引子文件均为**稳定文件（无日期，原地更新）**，描述当前支持的真相。

### `docs/index.md`

**用途**：文档路由器。目录表 + 每个目录的角色定义 + 核心原则 + 推荐默认路径。

**什么时候看**：不确定该从哪个目录入手时；想理解目录之间的职责分工时。

### `docs/architecture/`

**用途**：技术层吸引子——横切技术结构和模块边界的真相。

| 文件 | 内容 |
|------|------|
| `project-vision.md` | 长期产品方向、非目标、必须永远成立的约束 |
| `system-baseline.md` | 当前技术栈、部署形态、依赖方向、禁止的反模式 |
| `module-boundaries.md` | 各模块职责划分和接口契约 |

**什么时候看**：选型时、改技术栈时、跨模块开发时、理解系统边界时。

### `docs/design/`

**用途**：应用层吸引子——产品行为、界面、角色、工作流的真相。

| 文件 | 内容 |
|------|------|
| `app-overview.md` | 主要界面/页面、主导航、核心工作流、关键领域对象 |
| `feature-inventory.md` | 当前支持的功能清单 |
| `roles-and-permissions.md` | 角色定义及权限模型 |

**什么时候看**：加新页面/功能时、改权限时、理解现有应用行为时。

**与 `architecture/` 的区别**：`design/` 负责"用户看到什么、能做什么"，`architecture/` 负责"技术怎么支撑"。需要时交叉引用，但不合并。

### `docs/requirements/`（稳定部分）

**用途**：当前里程碑范围的真相。

**稳定文件**：`product-scope.md`（产品范围和约束）、`mvp.md`（MVP 边界和退出标准）。

**时效文件**：带日期的一次性需求综合文件（如 `2026-06-08-用户管理.md`）属于执行链，不属吸引子。

**什么时候看**：确定第一版要做什么/不做什么时。

## 执行链（流水线）

执行链是一条首尾相连的管道，从原始输入到代码落地到记录。

### `docs/input/`

**用途**：原始外部输入的存放地。PM 笔记、原型截图说明、卡片集文档、文章摘录、复制的用户反馈。

**规则**：保持原材料原样，**不要加工**。保留模糊和矛盾。

**什么时候看**：需求不够清晰时回溯源材料；PM 给了新文档时放入。

**入口**：`00-input-processing-guide.md`

### `docs/discussions/`

**用途**：可选。当原始输入模糊到不能直接写需求时，在此提出待澄清问题、记录假设、等待确认项。

**规则**：文件名带日期（`YYYY-MM-DD-topic.md`）。仅材料不清晰时使用。

**入口**：`00-discussion-writing-guide.md`

### `docs/requirements/`（时效部分）

**用途**：把澄清后的输入转化为**可实施的需求**——有具体验收标准、明确边界、无阻塞问题。

**规则**：一次性文件带日期（`YYYY-MM-DD-feature-name.md`）。需求没有验收标准 = 还没准备好实施。

**入口**：`00-requirement-synthesis-guide.md`

### `docs/backlog/`

**用途**：候选工作项优先级列表。每个项标注需求路径、所有者文档、计划、状态、AI 自主权标签、阻塞项。

**状态流转**：`idea` → `needs-requirement` → `needs-design` → `ready` → `in-progress` → `done`

**关键规则**：AI 只能将行降级（ready → blocked），不能升级，不能自行清除阻塞项。

**什么时候看**：每个功能完成后选下一个时；不确定是否可开始写代码时。

### `docs/plans/`

**用途**：非简单工作的执行计划。当工作涉及多模块/跨会话/API 变更/数据模型变更/受保护区域时必须创建。

**内容**：当前基线、目标/非目标、分阶段执行、退出标准、关闭关卡、技能选择记录。

**强制要求**：创建的计划必须经过**计划审计（实现前独立验证）**和**关闭审计（完成后独立验证）**。

**入口**：`00-plan-authoring-and-execution-guide.md`

### `docs/audits/`

**用途**：审计方法论 + 审计记录。独立质疑文档、计划、实现成果。

**强制项**：对已创建的计划，计划审计和关闭审计是强制性的。

**审计类型**：文档审计、计划审计、关闭审计。必要时叠加多维度审计或开放式审计。

**入口**：`00-audit-execution-guide.md`

### `docs/testing/`

**用途**：可选。手动测试笔记、探索性测试记录、已知良好基线。

**与 `lab/` 的区别**：`testing/` 记录"测了什么、结果是什么"，`lab/` 记录"环境怎么搭"。

### `docs/logs/`

**用途**：逐日实现记忆。按 `{year}/{month}-{day}.md` 组织，追记今天做了什么、做了什么决策、下一步是什么。

**规则**：顶部追加（倒序时间），保持简短，链接到相关文档路径。不是架构真相。

**入口**：`00-log-writing-guide.md`

### `docs/examples/`

**用途**：可复制的文件骨架。需求、讨论、计划、审计、回顾各一份示例 + 完整小应用演练。

**规则**：复制并重命名后填充，不要直接修改原文件。

## 经验库（沉淀物）

被动维护，只追加。不参与执行链，不触发审计或验证。

### `docs/skills/`

**用途**：可复用的工作方法模板（审计提示、检查清单、诊断提示）。11 个预设技能。

**关键原则**：技能是方法选择器，不是需求/设计的替代品。必须通过 `AGENTS.md` 和所有者文档路由后才能选用。

**什么时候看**：进行非简单任务前，找有没有匹配的工作方法。

### `docs/lessons/`

**用途**：从重复失败中提取的持久工程教训。按编号归档，如 `01-xxx.md`。

**与 `notes/` 的区别**：`lessons/` 是"摔出来的教训"（被动、痛感），`notes/` 是"学到的东西"（主动、正面）。

### `docs/notes/`

**用途**：主动整理的学习笔记和知识积累。如理解某个库的心得、研究某个模式的小结。

**规则**：不放原始书签、聊天记录复制、未消化的文章链接。稳定文件名，按主题组织。

### `docs/lab/`

**用途**：测试/实验环境搭建说明。数据库启动、mock 配置、种子数据脚本等。

**与 `testing/` 的区别**：`lab/` 放环境配方，`testing/` 放测试执行记录。

### `docs/analysis/`

**用途**：技术调研、方案权衡、被否决选项的理由记录。

**规则**：文件名带日期。当调研结果影响架构基线时，把结论移入 `architecture/`。

### `docs/bugs/`

**用途**：复杂回归的根因分析。非明显 Bug 才记录，小修小补不在此。

**规则**：可用编号命名（`01-xxx.md`）或带日期命名。

### `docs/retrospectives/`

**用途**：实施后的差距分析。回答"为什么交付没达标？缺了什么输入？哪个假设在实现时失败了？"

**规则**：文件名带日期。是分析，不是甩锅。

## 辅助目录

### `docs/archive/`

**用途**：由人类决定移至此处的不活跃文档。AI 不得自动归档。

**规则**：当文件在原始路径找不到时，来此查找。保留原始相对路径。

### `docs/articles/`

**用途**：面向外部的说明性方法论文章（AGE vs 规约驱动开发等）。不是日常执行的操作来源。

**规则**：当文章与所有者文档出现分歧时，以所有者文档为准。

### `tools/`

**用途**：独立 pnpm 子项目。CLI 检查工具：文档代码锚点检查、大文件检查、乱码检查、重复检查、代码统计、静态审计扫描。

**常见命令**：`pnpm check`、`pnpm stats`、`pnpm audit:suspects`
