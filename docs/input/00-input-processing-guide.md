# 输入处理指南

## 目的

本指南说明如何在原始来源材料转化为需求之前对其进行处理。

## 规则

当原始资料中仍混杂以下内容时，不要直接让 AI 从大规模原始输入转储中编写代码：

- 业务目标
- UI 示例
- 实现猜测
- 缺失的假设
- 未完全确定的范围

## 推荐流程

1. 将原始材料存储在 `docs/input/` 中。
2. 标记来源类型：PM 笔记、卡片集文档、原型、文章或混合来源。
3. 将未解决的问题写入 `docs/discussions/`。
4. 将综合结果写入 `docs/requirements/`。

## 有用的来源分类

- `source-pm-*.md` - 产品经理笔记
- `source-prototype-*.md` - 原型解读
- `source-cardset-*.md` - 卡片集或结构化需求文档
- `source-article-*.md` - 外部文章或参考资料

## 注意

结构良好的来源资料虽有用，但仍可能无法回答：

- 当前迭代的实际范围边界
- 业务决策所需的领域判断
- 哪些交互是核心功能，哪些是可选功能
- 原型是否足够完整，可以直接基于它进行构建

## 文件头约定

放入此目录的每个输入文件应以以下头部信息开头：

    status: new | supplement | supersedes <filename>
    processed: pending | partial | done

- `status` 描述文件与其他输入的关系：
  - `new` — 不替代现有文件的独立新输入
  - `supplement` — 为现有输入文件提供补充上下文
  - `supersedes <filename>` — 取代之前的输入文件；指定的文件现已过时
- `processed` 跟踪此输入是否已被消费：
  - `pending` — 尚未处理；代理应拾取此文件
  - `partial` — 部分问题已解决，但仍有未解决项
  - `done` — 已完全处理为需求或讨论；代理可跳过此文件

如果人工放置文件时未包含此头部信息，AI 应：

1. 阅读文件内容。
2. 根据内容和上下文推断状态。
3. 添加推断出的头部信息，并注明为 AI 推断。
4. 继续处理。
