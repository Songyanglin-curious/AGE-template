## 2026-06-02：强化强制性日志记录规则

### 背景

模板原本有 `docs/logs/` 和 `00-log-writing-guide.md`，但 AGENTS.md 只说“保持日志简短、注明日期且仅可追加”，没有强制触发条件。在实践中（nop-chaos-flux），AI 从未写入日志，因为没有条件强制要求。nop-chaos-flux 不得不自行添加“文档维护”章节，其中包含明确的强制触发条件、验证清单以及全绿基线记录规则。

### 决策

- 更新了 AGENTS.md 操作规则 7：添加了强制触发条件“完成任何重要代码更改后，必须更新每日开发日志”
- 向 AGENTS.md 添加了 `Docs Maintenance` 章节，包含两个强制项（日志更新 + 负责人文档更新）和全绿基线记录规则
- 将 `docs/logs/00-log-writing-guide.md` 从一个 19 行的骨架扩展为完整指南，包含目的、规则（6 项）、路径约定、条目格式示例以及分步的“添加新条目”说明
- 向 `START-HERE-after-copy.md` 必备清单中添加了日志目录初始化步骤

### 理由

没有强制触发条件和具体的格式指南，日志规则对 AI 代理来说形同虚设。模板虽然定义了目录结构和占位指南，但 AGENTS.md 和该指南都没有告诉代理何时或如何写入条目。nop-chaos-flux 证明了带有 MUST 级别触发条件和全绿记录规则的“文档维护”章节能让日志记录真正落实。

### 后果

复制的项目从一开始就具备可强制执行的日志规则。扩展后的 `00-log-writing-guide.md` 提供了足够的格式细节，代理无需猜测就能编写结构良好的条目。START-HERE 清单确保日志目录结构在首次编码会话之前就已就绪。

## 2026-05-29：添加可复用的工程技能和可选工具工作区

### 背景

模板现在包含更广泛的可复用工程层面：错误诊断、代码质量审计、重构发现、重构执行、根级文本文件默认设置，以及用于通用检查的仓库本地 `tools/` 工作区。

### 决策

- 添加了 `docs/skills/bug-diagnosis-prompt.md`
- 添加了 `docs/skills/code-quality-audit-prompt.md`
- 添加了 `docs/skills/code-refactor-discovery-prompt.md`
- 添加了 `docs/skills/code-refactor-prompt.md`
- 更新了 `docs/skills/README.md`
- 添加了 `.editorconfig`、`.gitattributes` 和 `.gitignore`
- 添加了 `tools/` 作为可选 pnpm 子项目，用于可复用的仓库本地工具

### 理由

做出此更改是因为模板虽然已定义了文档驱动的工作流，但缺乏针对路由后反复出现的三种常见执行模式的可复用方法：证明根本原因、审计修改后的代码以评估实际风险，以及在不悄然改变行为的前提下重构代码。

仓库还需要为复制的项目提供一个小的默认质量基线。根级文本规范化可减少可避免的差异，而将基于 Node 的工具放在 `tools/` 内可保持模板的跨技术栈定位，避免将仓库根目录变为仅限 Node 的项目。

### 后果

复制的项目可以立即采用更广泛的可复用工程方法，并且它们有一个现成的地方来发展轻量级的仓库本地检查，而无需将整个模板耦合到单一运行时技术栈。

## 2026-05-29：添加索引路由审计技能

### 背景

现有的审计技能涵盖了文档质量和计划/决策审计，但没有一项能验证索引树（`docs/index.md` 加上每个目录的 README）是否真正将读者引导至正确位置。

### 决策

- 添加了 `docs/skills/index-routing-audit-prompt.md`

### 理由

做出此更改是因为路由质量是一个反复出现的模板风险：复制的项目可能拥有正确的负责人文档，但仍无法引导代理和维护者找到正确的下一份工件。

路由失败通常是情境性的：索引条目可能在技术上正确，但仍然无法满足特定角色的需求。简单的清单会遗漏这一点。该提示使用三个层次——覆盖表、基于角色的路由测试、结构检查——从而同时评估机械正确性和情境可达性。
