# 需求梳理指南

## 目的

使用本指南将原始输入转化为可立即开发的需求。

## 输入材料

- `docs/input/` 原始素材
- `docs/discussions/` 澄清备注
- 已有的 `docs/design/` 和 `docs/architecture/` 负责人文档

## 输出预期

需求文件应清晰陈述：

- 目标
- 范围
- 非目标
- 核心用户流程
- 业务规则
- 权限或角色影响
- 已知边缘情况
- 待解决问题
- 验收标准

## 规则

勿以华丽辞藻掩盖未解决的歧义。务必明确记录。

## 可开发就绪门槛

只有当以下条件全部满足时，需求才算可开发就绪：

- [ ] 范围内行为足够具体，无需猜测用户可见行为即可构建
- [ ] 非目标已明确
- [ ] 主要用户流程已描述
- [ ] 角色/权限已覆盖或明确不相关
- [ ] 数据/模型影响已覆盖或明确不相关
- [ ] API/集成影响已覆盖或明确不相关
- [ ] 用户可见的空状态、加载状态、错误状态、权限拒绝状态已覆盖
- [ ] 验收标准可测试
- [ ] 待决问题要么不阻塞开发，要么明确阻塞开发

如果某个待决问题会影响用户可见行为、数据/模型结构、API 行为、认证/权限行为或外部集成行为，则暂停并请求澄清。

## 最小需求骨架

```md
# 特性：<name>

## 目标

## 范围内

## 范围外

## 主要用户流程

## 业务规则

## 角色 / 权限

## 边界情况

## 待解决问题
```
